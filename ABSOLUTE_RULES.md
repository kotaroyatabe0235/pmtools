このルールは、いかなる状況においても、必ず守られるべきルールで逸脱は許されない。また、このルールの変更は人間によってのみ行われるため、いかなる場合も変更は許容されない。
このファイルを参照したら、すぐさま絶対順守ルールとして自身のメモリに必ず保存する。

# ルール
* 変更はmainブランチからfeatureブランチを作成し、人間に変更のPR を承認させる
* PRは、人間から指摘がある旨とPRのリンクがチャットされる。これを受け取ったらgh api repos/$OWNER/$REPO/pulls/$PR_NUMBER/commentsで取得して確認し、すべて対応したらcommit&pushせよ
* featureの単位は、50〜100stepを上限とし、意味ある単位に計画する
* 作業の状況は逐一標準出力するとともに、コードはコードと同量のコメントで実装の意図や背景を記載する

# 案件
## 案件概要
* microsoft projectを実装する

## 技術要素
* Spring Boot + thimeleaf + jQuery
* postgreSQL
* composeでコンテナとして稼働
上記以外の技術要素の採用は認めない。

## 開発スタイル
* TDD

## 計画
* 設計
* 必要ルールの洗い出しと人間による定義
* テスト製造
* 実装
* READMEの作成
全体において、各フェーズの最初と最後に、計画・または再計画を行う。

## ディレクトリ
root
>docs            : design documentを配置するディレクトリ。この配下にmd形式でドキュメントを配置。これ以上のディレクトリ作成は許可しない。
>rules           : 各種ルールを配置するディレクトリ（配下ファイルは人間のみ変更を許可）。ルールはmd形式で人間が配置。これ以上のディレクトリ作成は許可しない。
>src             : 実態ソースを配置するディレクトリ。自由にファイルを配置して良い。内部のディレクトリ作成も任意。
>tools           : dockerfileやCIに関する設定ファイル等を配置するディレクトリ。内部のディレクトリ作成も任意。
>README.md       : 単一のmdファイル。使用や使い方などを最終的に取りまとめる。
>ABSOLUTE_RULE.md: 開発における絶対的なルールやこのリポジトリの目的などを記載する。(人間のみ編集可能)
>PLAN.md         : 開発の計画を記載する
